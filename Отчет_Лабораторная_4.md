# Лабораторная работа №4

Тема №3: «Реализация системы оценки кода на плагиат с использованием LLM».

## 1. Цель работы

Получить практические навыки разработки, интеграции и тестирования дополнительной бизнес-логики в составе LabServer: подсистемы оценки схожести программного кода с использованием LLM.

## 2. Задачи

1. Реализовать и интегрировать в LabServer систему оценки кода на плагиат с использованием LLM.
2. Использовать локальную модель Ollama `gemma2:2b` как LLM-анализатор.
3. Реализовать обучение на лету: после каждой проверки сохранять только новый измененный файл в накопительный корпус.
4. Вынести компоненты в отдельные контейнеры Docker Compose.
5. Отобразить процент плагиата во фронтенде LabServer.
6. Реализовать функциональное и интеграционное тестирование дополнительной бизнес-логики.

## 3. Ход работы

### 3.1. Разработанные компоненты

Добавлены общие модели результата проверки:

- `Shared/Models/Plagiarism/CodeFileModel.cs` - представление файла исходного кода.
- `Shared/Models/Plagiarism/PlagiarismCheckResponseModel.cs` - итог проверки: признак плагиата, максимальная схожесть, порог, сообщение и список ближайших совпадений.
- `Shared/Models/Plagiarism/PlagiarismMatchModel.cs` - детализация совпадения с конкретным эталонным файлом.

Добавлена серверная бизнес-логика:

- `CodeNormalizer` удаляет комментарии из C/C++/C#-подобного кода, сохраняя строки и символьные литералы, затем нормализует токены.
- `TokenSimilarity` считает структурную схожесть по токенным шинглам.
- `OllamaGemmaAnalyzer` обращается к Ollama API `http://localhost:11434/api/generate` и использует модель `gemma2:2b`.
- `PlagiarismAnalyzer` объединяет токенную оценку и оценку LLM в итоговую метрику.
- `SubmissionCodeExtractor` извлекает исходные файлы из zip-архива репозитория.
- `ReferenceCodeProvider` загружает базовые эталоны и накопленные примеры из обучающего корпуса.
- `OnlineLearningCorpus` сохраняет проверенные новые файлы в persistent-корпус и использует их как обучающие примеры в следующих проверках.

Интеграция в LabServer:

- Добавлен endpoint тестового сервера `POST /test/plagiarism/schedule` и `POST /test/plagiarism/getresult`.
- `TestRunner` теперь скачивает zip-архив репозитория из GitLab через `/projects/{id}/repository/archive.zip?sha={commit}` и передает его в тестовый сервер вместо прежней заглушки `TODO`.
- `TestRunner` получает список измененных файлов merge request и передает его в `ChangedFilePaths`.
- `PlagiarismTestController` фильтрует архив по `ChangedFilePaths`, поэтому в анализ и обучение попадают только новые/измененные файлы, а не все предыдущие файлы репозитория.
- `StudentLabSubmissionData` теперь отдает связанные `TestRuns`, поэтому фронтенд может показать сообщение и процент проверки на плагиат.
- В админском интерфейсе и студенческом dashboard добавлена колонка `Плагиат` с процентом схожести, цветовым индикатором и progress bar.
- В `appsettings.json` добавлена секция `Plagiarism` с порогом `0.75`, директориями эталонов и настройками Ollama.
- В `compose.yml` добавлены отдельные сервисы `ollama`, `ollama-init` и `plagiarism-server`, а также volume `plagiarism-learning` для накопленного корпуса.
- В `compose.yml` nginx подключен к актуальным Angular build-артефактам `web/admin/dist/admin` и `web/dashboard/dist/dashboard`.

### 3.2. Функциональные и интеграционные тесты

Добавлены тесты:

- `CodeNormalizerTests` - проверяет удаление комментариев без повреждения строковых литералов и нормализацию идентификаторов.
- `PlagiarismAnalyzerTests` - проверяет выявление структурно похожего кода и обработку пустой посылки.
- `PlagiarismTestControllerTests` - проверяет полный цикл тестового сервера: `schedule` принимает zip-архив, анализирует код, `getresult` возвращает результат проверки.
- Дополнительно проверено, что обучение получает только файл из `ChangedFilePaths`.

Результат запуска:

```text
Passed: 9, Failed: 0, Skipped: 0
```

Команда:

```bash
~/.dotnet/dotnet test labserveroriginal/LabServer.sln
```

### 3.3. Пример работы интегрированной системы

1. В LabServer для теста лабораторной работы указывается URL:

```text
http://localhost:<port>/test/plagiarism
```

2. `TestRunner` получает активную сдачу, скачивает архив GitLab и вызывает:

```http
POST /test/plagiarism/schedule
```

3. Контроллер извлекает из архива только файлы из `ChangedFilePaths`, сравнивает их с `reference.c` и ранее выученными примерами, вызывает `gemma2:2b`, сохраняет результат.

4. При запросе:

```http
POST /test/plagiarism/getresult
```

возвращается сообщение вида:

```text
Possible plagiarism: closest file is reference.c. Similarity=91.2%, threshold=75.0%, best match=reference.c: 91.2%
```

Если максимальная схожесть ниже порога, результат теста считается пройденным.

Проверка на файлах из директории выполнялась последовательно: `prog1.c`, `prog2.c`, `prog3.c`, `prog4.c`, `prog5.c`. Каждый запрос содержал zip только с одним текущим файлом. После каждого успешного анализа размер обучающего корпуса увеличивался на один файл: после пяти проверок сохранено пять записей.

## 4. Выводы

В LabServer интегрирована подсистема оценки кода на плагиат. Проверка учитывает как структурную схожесть нормализованного кода, так и оценку LLM `gemma2:2b`, что позволяет выявлять решения с переименованными переменными и незначительно измененной структурой. Обучение на лету реализовано через persistent-корпус новых проверенных файлов: последующие проверки используют ранее сохраненные решения как обучающие примеры. Реализованы функциональные и интеграционные тесты, подтверждающие корректность ключевых сценариев.
