# Ручное тестирование через веб-интерфейс

Документ описывает, как вручную проверить работу лабораторной через браузер: запуск локального стенда, проверку админки, проверку dashboard студента и контроль отображения процента плагиата.

## 1. Запуск стенда

Из корня проекта:

```bash
cd /Users/alexey/CLionProjects/trspo-3lab
make up
```

Проверить состояние контейнеров:

```bash
make ps
```

Ожидаемо должны быть запущены:

- `db`
- `labserver`
- `plagiarism-server`
- `webserver`

Локальные адреса:

```text
Админка:             http://localhost:8080
Dashboard студента:  http://localhost:8081
Plagiarism API:      http://localhost:5289/test/plagiarism
```

## 2. Первичная проверка доступности UI

1. Открыть `http://localhost:8080`.
2. Убедиться, что загружается админский интерфейс LabServer.
3. Открыть `http://localhost:8081`.
4. Убедиться, что загружается страница входа студента по токену.

Если страницы не открываются:

```bash
make ps
make logs-web
make logs-server
```

## 3. Регистрация администратора

Если база данных новая:

1. Открыть `http://localhost:8080/register`.
2. Ввести email и пароль.
3. Нажать кнопку регистрации.
4. Первый зарегистрированный пользователь автоматически получает роль `Administrator`.
5. Перейти на `http://localhost:8080/login`.
6. Войти под созданной учетной записью.

После входа должны быть доступны разделы:

- `Группы`
- `Курсы`
- `Тесты`
- `Пользователи`

## 4. Настройка теста на плагиат в UI

1. Открыть раздел `Тесты`.
2. Нажать `Создать`.
3. Заполнить:

```text
Название: Plagiarism Gemma 2B
URL-адрес сервера тестирования: http://plagiarism-server:5288/test/plagiarism
```

4. Сохранить тест.
5. Убедиться, что тест появился в таблице.

Важно: URL указывается внутренний docker-compose URL, потому что `TestRunner` вызывает тестовый сервер из контейнера `labserver`.

## 5. Подготовка лабораторной через UI

В админке:

1. Открыть раздел `Курсы`.
2. Создать курс.
3. Открыть задания курса.
4. Создать лабораторную работу.
5. Открыть раздел `Группы`.
6. Создать группу.
7. Открыть студентов группы.
8. Добавить студента.
9. Открыть курсы группы.
10. Назначить созданный курс группе.
11. Открыть страницу прогресса группы по курсу.
12. Открыть лабораторную для студента.

На странице прогресса у студента должна появиться карточка лабораторной.

## 6. Важное ограничение текущего frontend

Текущий frontend позволяет создать тест в разделе `Тесты`, но не содержит отдельного UI-экрана для привязки теста к конкретной лабораторной работе.

Для полной проверки через merge request связь `лабораторная -> Plagiarism Gemma 2B` должна уже существовать в базе данных. Если связи нет, `TestRunner` не будет запускать проверку плагиата для MR, и на фронте будет отображаться `Нет данных`.

Признак корректной привязки: после создания MR для лабораторной появляется `TestRun`, а в сообщении теста есть строка вида:

```text
Similarity=17,4 %
```

Именно это значение frontend показывает в колонке `Плагиат`.

## 7. Полный сценарий через GitLab и MR

Для полного сценария нужен GitLab-контейнер и включенные фоновые сервисы:

```bash
make up-full
```

После запуска:

1. Открыть админку: `http://localhost:8080`.
2. Настроить курс, группу, студента и лабораторную.
3. Выполнить GitLab-синхронизацию там, где в UI есть кнопка с иконкой GitLab.
4. Перейти в GitLab: `http://localhost:8082`.
5. Открыть проект студента.
6. Добавить или изменить C-файл, например `prog1.c`.
7. Создать merge request.
8. Подождать 10-30 секунд, пока `TestRunner` обработает MR.
9. Вернуться в админку.
10. Открыть страницу студента по лабораторной.

Ожидаемый результат в таблице `Сдача лабораторной`:

- есть строка с MR;
- есть колонка `Плагиат`;
- в колонке отображается процент, например `17,4 %`;
- под процентом виден progress bar;
- при высоком проценте индикатор красный, при среднем желтый, при низком зеленый.

## 8. Проверка в dashboard студента

Если известен dashboard token студента:

1. Открыть `http://localhost:8081`.
2. Ввести token.
3. Открыть список лабораторных.
4. Найти проверяемую лабораторную.

Ожидаемый результат:

- в таблице есть колонка `Плагиат`;
- для последней сдачи отображается процент;
- если проверка еще не прошла, отображается `Нет данных`.

## 9. Что считается успешным тестированием

Сценарий считается пройденным, если:

1. Админка открывается на `http://localhost:8080`.
2. Dashboard студента открывается на `http://localhost:8081`.
3. В админке можно открыть страницу конкретной лабораторной студента.
4. После обработки MR в таблице сдач появляется процент плагиата.
5. Процент совпадает с `Similarity=...` из результата теста.
6. При обновлении страницы процент не пропадает.
7. В dashboard студента процент также отображается для соответствующей лабораторной.

## 10. Диагностика

Если в UI отображается `Нет данных`:

```bash
make logs-server
make logs-plagiarism
```

Проверить:

- создан ли MR;
- есть ли измененный C-файл в MR;
- привязан ли тест плагиата к лабораторной;
- указан ли URL теста `http://plagiarism-server:5288/test/plagiarism`;
- запущены ли фоновые сервисы, то есть используется `make up-full`, а не только быстрый `make up`;
- доступна ли модель `gemma2:2b`.

Проверить модель:

```bash
ollama list
```

В списке должна быть:

```text
gemma2:2b
```

Если нужно быстро убедиться, что backend-часть проверки работает:

```bash
make smoke
```

Эта команда не заменяет ручную UI-проверку, но подтверждает, что `plagiarism-server`, Ollama и обучение на лету работают.
